# Anees Health Platform — CTO Technical Due-Diligence Audit

> **Date:** 2026-07-01 · **Auditor:** Acting CTO review · **Method:** Direct source-code audit across 14 dimensions (36 analyst passes), every serious finding adversarially re-verified against the code; 4 findings dismissed as false alarms. Overall maturity **≈ 59/100**.

---

## 1. Bottom line

You have built an **impressively engineered application sitting on a dangerously under-operated platform.**

The *product code* — the clinical EHR, the FHIR/Medplum integration, the authorization engine, the bilingual frontend — is genuinely senior-grade work, well above what a solo founder typically ships. The *operational envelope* around it — monitoring, disaster recovery, testing of the money/PHI paths, tenant isolation, and staffing — is where the real risk lives.

**Nothing here is an active breach today.** But there is a specific cluster of security, financial-integrity, and continuity gaps that must be closed **before the hospital partner goes live**, and one existential business risk (bus factor of one) that should be treated as the number-one funding priority.

## 3. The one critical (P0) issue

### Bus factor = 1 — the entire regulated PHI platform is one person
All 167 commits are from a single author (git identity is still the placeholder `Your Name / your-email@example.com`), with no human code review and no merge/PR flow. The strategy doc itself mandates *"no single engineer ships PHI-touching code alone"* and *"code review is mandatory even for the founder"* — neither happens. Combined with **no infrastructure-as-code in the repo** (no Dockerfile, compose file, or Caddyfile — the production topology exists only on the VPS and in your head), a single event (illness, dispute, VPS loss) could make the platform unoperable and clinical data unrecoverable.

**Do first:** (1) hire the "senior full-stack engineer — now" role your own plan lists; (2) institute real review even at team-of-two; (3) fix the git identity today; (4) commit the production topology as IaC and run one real restore drill. **Do not sign a second hospital tenant until a second engineer can operate the stack.**

## 4. Systemic themes (the root causes behind most findings)

Most of the ~23 high findings trace back to **five root causes**. Fix the root, and many findings close together.

### Theme A — "You can't see what's happening" (Observability/DR, score 31)
- **Sentry is installed but switched off** — DSN-gated, inactive in production. An unhandled error, PHI-leak bug, crash loop, or payment failure produces **no alert**; the only detection path is a human tailing `pm2 logs` on the VPS.
- **Logs are console-only and never aggregated** — critical audit-failure markers (`AUDIT_WRITE_FAILED`, `AUDIT_FHIR_MIRROR_FAILED`) go to stdout on one box, unmonitored.
- **Single-VPS SPOF + un-drilled backups** — app, Postgres, and Medplum share one Hostinger host. Backups/PITR are described as "target"; only a nightly `pg_dump` exists, and no restore has been drilled. RTO/RPO are aspirational.
- **No externally-pollable health endpoint** — the only health probe requires admin auth, so no uptime monitor can watch the platform.
- **No durable job queue** — background work (audit mirror, malware rescan, portal invite) runs on `after()`/cron; work in flight is lost on deploy/crash with only a possible log line.

### Theme B — "The safety nets are built but not switched on"
A recurring, almost signature pattern in this codebase: a strong mechanism is written, then not adopted.
- **Tenant isolation:** a forget-proof auto-scoping Prisma client (`tenant-prisma.ts`) exists — with **zero call sites**. Real isolation rides on a hand-written helper + a heuristic CI guard that has a documented same-file bypass and 15 grandfathered violations. So "multi-tenant from day one" is single-tenant in practice.
- **Tamper-evident audit chain:** the hash chain protects clinical writes but **not** the highest-value events — PHI document downloads, logins/logouts, restricted-record reads, and **every payment/invoice/refund row** are written with raw `auditLog.create()` that bypasses the chain and the FHIR mirror. An insider with DB access could delete a PHI-download or refund record and the verifier still reports "intact."
- **Observability seam:** fully wired to all 5 error boundaries — just missing the DSN env vars.

### Theme C — "Tests don't cover the money, PHI, or authz paths" (Testing, score 38)
The guards (RBAC matrix, tenant-scope, CSS, migration drift) are good, but the actual risk surface is untested: the **Kashier payment webhook**, **all 33+ EHR server actions** (the real authz/IDOR enforcement), the **22-state visit machine**, and the **OTP/register/reset** flows have zero tests. No E2E. No coverage threshold. The audit-chain verifier isn't wired into CI.

### Theme D — "Money integrity has holes" (Payments, score 54)
- **The ledger books the expected price, never the amount actually charged.** On a partial capture or gateway adjustment, `Payment.amountEgp` still records the booked figure; the real amount survives only as a text note.
- **Reconciliation treats a 100× over/undercharge as a match** (a deliberate unit-tolerance that was never tightened).
- **Two payment routes are unauthenticated** (`/payment/initiate`, `/payment/instapay`) — anyone with a booking reference can create Kashier sessions, issue invoices, or overwrite another customer's InstaPay sender/reference metadata (the ref is high-entropy, so this is broken-authorization, not trivial enumeration).
- **No webhook replay/nonce defense** — a replayed signed refund event can flip a paid booking to refunded.
- **`ProviderPayout` is a dead model** — clinician payouts are never computed or ledgered; the nursing dashboard aggregates an always-empty table.

### Theme E — "Docs describe a more finished platform than the code is"
Several "✅ done" claims don't survive inspection: DPL booking-consent capture (no code), the auto-tenant client ("done" but unused), insurance adjudication (a read-only page), "anyone can rebuild in <2h" (no IaC), and CLAUDE.md still says clinician is "physio-only" when full doctor + nurse workspaces ship live code touching PHI. This is a governance/procurement risk more than a code risk — a hospital security reviewer *will* find the gaps.

## 5. Dimension detail (verified findings, most-severe first)

### Auth / AuthZ / RBAC — 78 (strongest)
**Strengths:** genuine layered policy engine (`ehr-matrix → actions → can → enforce/pip`) actually enforced on every sampled server action; IDs re-resolved server-side (IDOR prevented); case-scope from live CareTeam; the prior task-IDOR is fixed; staff tokens are hashed/single-use/constant-time; deny-by-default edge gate.
- **P1 — Login endpoint has no brute-force/rate limiting.** `[...nextauth]` re-exports handlers verbatim; `bcrypt.compare` is the only throttle. Custom auth routes are rate-limited; the actual password-verification path is not. → Wrap `authorize()` with `checkRateLimit` + per-account lockout + audit.
- **P1 — Suspended staff / role-downgrades aren't enforced until the JWT expires (≤45 min).** `status` is checked only at login and not re-read on request; the edge gate trusts the stale token. → Add `status` to the PIP select, fail closed, re-check in the jwt callback.
- **P2** — Break-glass approval allows self-approval (no separation of duties). **P2** — reasoned restricted-read is self-issued (accountability, not prevention — verified as *intentional* break-glass-lite, case-scoped, 30-min TTL; a policy call, not a bug). **P2** — in-memory OTP store won't survive multi-instance.

### Frontend, UX, i18n & Accessibility — 76
**Strengths:** perfect EN/AR key parity (1765 keys each), all 5 error boundaries real and PHI-safe, RTL fully swept to logical CSS (guard at zero), design tokens enforced, high-quality loading/ARIA.
- **P1 — Document root `<html>` is hard-locked to `lang="en" dir="ltr"`** and never flips for Arabic (direction is set on an inner `<div>`). WCAG 3.1.1 failure; body-appended widgets/portals/toasts don't flip. → Set lang/dir on the real `<html>` per locale.
- **P2** — site-wide skip-link points to `#main-content`, missing on ~12 pages (doctors, about, contact, all auth) → dead anchor (WCAG 2.4.1). **P3** — mobile burger `aria-expanded` is static; SEO pages fork i18n with inline `isAr` ternaries; a few raw `<img>` on the doctor grid.

### Architecture & Code Quality — 72
**Strengths:** clean dependency direction (`lib/` never imports `features/`), exemplary admin-patient action split, low cruft, all error boundaries present.
- **P1 — Three clinician workspaces (physio/nurse/doctor) are copy-paste clones and have already diverged on a security line.** Only the doctor path strips a client-supplied `encounterId`; nurse and physio do not, so a nurse/physiotherapist can post a signed Observation/note bound to *another patient's* encounter reference. → Extract one shared, parameterised write-wrapper (always stripping `medplumPatientId` + `encounterId` + `assessmentEncounterId`).
- **P2** — CLAUDE.md materially understates what's built; dead `tenant-prisma.ts`. **P3** — inconsistent `/clinician` URL topology; oversized SEO content files (no size ratchet).

### Clinical Core (Medplum/FHIR) — 72
**Strengths:** correct LOINC/UCUM vitals + BP panel, discrete coded outcome measures with trend reader, correct FHIR clinical-status constraints, proper note-signing with attester + Provenance + optimistic concurrency, `entered-in-error` everywhere (never hard delete), real license/case-scope gating.
- **P1 — The 22-state visit machine has no transition-legality map.** Legality is enforced only by ad-hoc timestamp checks; the state write and the timestamp/status write happen in **separate transactions** (crash → permanent divergence), and `fromState` is read outside the transaction (TOCTOU → two concurrent transitions both commit). → Add an explicit `VALID_TRANSITIONS` map, move the writes into one transaction, use a conditional `where {id, state: expectedFrom}`.
- **P1 — `force=on` override bypasses every guard** (sequence, live-location, geofence, clinical-evidence) with **no role gate or break-glass token** — any clinician can check a visit out (triggering completion/payout) with no GPS and no documentation. It's audited as an override, but nothing *prevents* it. → Gate `force` behind medical_ops/admin or a `DestructiveApprovalToken`; forbid on check-out for non-supervisors.
- **P2** — `VisitLocationPing` / `VisitParticipant` / identity-confirmation / consent-reaffirmed / safety-clearance fields are **dead schema** (never written) despite advertising safety capabilities; set-status mutators lack optimistic concurrency (lost updates on meds/allergies). **P3** — labs store display unit as UCUM code; Encounter sorted by an unsupported param; pre/post pain share one LOINC.

### Dependencies & Supply Chain — 68
**Strengths:** committed integrity-checked lockfile, precisely pinned Node engines, strong CI, current core deps.
- **P1 — No automated dependency updates or audit gate; vulns drifted to 14 (1 critical, 2 high) unnoticed.** The scary ones are dev/build-only today, but there's *zero signal* when a runtime one lands. → Add Renovate/Dependabot + a `npm audit --audit-level=high --omit=dev` CI step.
- **P2** — `ts-node` shipped in production `dependencies`; `serialize-javascript` (build-time) and `react-slick`→jQuery pulled in for one carousel; `next-auth` on a pre-release beta with a caret. **P3** — Prisma pinned two majors behind (plan a deliberate 5→6→7).

### Database & Multi-Tenancy — 66
**Strengths:** write-path tenant enforcement is genuinely centralized in the policy engine; thorough indexing; correct Prisma singleton; safe parameterized raw SQL; state-machine ledger honored.
- **P1 — Tenant isolation is by convention + heuristic guard, not by construction** (the auto-scoping client is dead code; the guard has a same-file bypass and 15 baselined violations). → Adopt the auto-client or delete it; drive baseline to zero before tenant #2; add Postgres row-level security for the hospital pilot.
- **P1 — Cascade deletes wire regulatory tables to Patient/Visit lifecycle** — a hard `patient.delete()` would cascade-erase the EDA controlled-substance ledger, claims, prior-auths, coverage. Latent (no hard-delete today) but the schema doesn't enforce the never-hard-delete rule, and dev points at prod. → Change these FKs to `Restrict`.
- **P2** — no uniqueness on `Patient.phone`/`nationalId` (duplicate-identity hazard); `tenantId` is a plain string, not a FK; money/break-glass tables have loose refs; a migration is named `test`; newer models regress to free-text status columns.

### PHI & Data Protection — 62
**Strengths:** private R2 bucket with AES256, random-UUID keys, no public/presigned URL exposure; the document route is the strongest handler in the codebase (role + case-scope + ownership, fail-closed malware gate, checksum integrity, PHI-safe headers); session-scoped `getOwnPatientRecord`; no PHI or secrets in logs.
- **P1 — PHI-download / access-denied audit records bypass the tamper-evident hash chain and are swallowed on failure** (raw `auditLog.create` in an empty `catch`). The exact "who exported which patient's file" records an auditor cares about most are neither chained nor guaranteed to persist. → Route through `recordAudit`; treat post-cutover null-hash rows as chain breaks.
- **P2** — `getPrivateMedicalObject` buffers the **whole file into memory** and the route **re-hashes it in memory** on every download (OOM/DoS on the shared VPS; caps servable file size). → Stream, or issue short-TTL signed URLs. **P2** — bucket privacy/encryption depend on undocumented external config with no code-level assertion.
- *(Dismissed: "malware scanner fails open to mock_clean" — refuted; `assertProductionReadiness()` at boot refuses to start in production if the scanner is mock/unset, and the serve path is fail-closed.)*

### API Routes & Input Validation — 62
**Strengths:** security-critical routes use fail-closed rate limiting; webhook HMAC is correct; prices/discounts always recomputed server-side; document route excellent.
- **P1 — `/bookings/payment/initiate` and `/bookings/payment/instapay` are fully unauthenticated.** With a booking ref, a caller can create Kashier sessions, (re)issue that booking's invoice, and overwrite InstaPay sender/reference or reset a failed booking to pending. → Bind to the booking owner via session or a short-lived signed booking token; make instapay metadata write-once; add CORS to initiate.
- **P2** — in-memory OTP store breaks on multi-instance (behind durable rate limits, so P2); non-constant-time secret compares on `pwa/send` and the scan route; validation hand-rolled/inconsistent (only one route uses Zod; `register` allows 8-char passwords vs the 12-char policy elsewhere); no request-body size ceiling. **P3** — mild account enumeration on register; a couple of authenticated GETs lack rate limiting/caching.

### Performance & Scalability — 58
**Strengths:** the heaviest page batches ~26 reads into one `Promise.allSettled` with tab-aware lazy loading; bounded FHIR pagination; good image config; resilient Medplum singleton.
- **P1 — Patient directory has no pagination/search and caps at the 20 globally-newest patients before tenant/case filtering** — a case-scoped clinician may see few or none of *their* patients and can't reach the rest. → Push filtering into the Medplum query with paging + search.
- **P1 — N+1: up to ~80 round-trips hydrating per-visit workflow state** on the patient-detail loader (against the shared remote prod DB). → Add `state` to the batch select; one `findMany({visitId: {in: […]}})` instead of the loop.
- **P1 — Default `next-pwa` runtime caching persists `/api/*` and portal RSC (PHI) to browser Cache Storage for 24h** — readable on a shared clinician tablet after logout; the doc route's `no-store` is ineffective against Workbox. → Ship an explicit cache list that excludes all authenticated paths.
- **P2** — no app-layer request-dedup caching; no Prisma pool limit vs shared Postgres; compliance dashboard has no `action` index/pagination. **P3** — Bootstrap CSS ships to the whole public site; four serial gate round-trips precede the batch.

### Payments & Billing — 54
See Theme D. **Strengths:** `Decimal(10,2)` money (no float drift), correct constant-time HMAC + fail-closed webhook, idempotent transactional upserts, server-side discount math with race-safe promocode redemption, role-gated tenant-scoped admin billing.
- **P1** — ledger books expected-not-actual amount. **P1** — financial audit rows bypass the hash chain. **P2** — 100× reconciliation tolerance; no webhook replay defense; HMAC signs only the attacker-supplied field-set; dead `ProviderPayout`; EGP hardcoded (blocks MENA). **P3** — refunds are status-only (no gateway call), status is free-text.

### Compliance & Governance — 48
**Strengths:** honest, control-mapped docs; dual-store audit with sound retry/non-swallow semantics; real two-party break-glass + weekly auto-escalation; restricted-tier masking that actually redacts; consent-scoped caregiver access.
- **P1** — the highest-value audit events (PHI download, login/logout, restricted read) bypass the hash chain (see PHI). **P1** — documented DPL consent capture (booking consent, cross-border transfer, data-subject export) **has no implementation** while the doc tells procurement it's done. → Add versioned consent capture + a DSAR export path; correct the doc.
- **P2** — restricted-tier text detection is **English-only** (verified: the *coded* ICD-10 path works for Arabic; the gap is uncoded free-text Arabic notes — real but narrow); logout audit is best-effort client-triggered; caregiver consent implements 5 of 8 documented scopes and defaults to `profile+visits` instead of deny.
- *(Dismissed: "reasoned restricted access is a self-service backdoor" — refuted; it's case-scoped, 30-min TTL, dual-audited break-glass-lite, by design.)*

### Product & Strategy Readiness — 42
**Strengths:** genuinely excellent, honest strategy docs; sound FHIR/Medplum + Postgres core bet; real anti-vendor-lock-in discipline; CI guardrails rare in solo projects.
- **P0** — bus factor 1 (see §3). **P1** — no IaC in repo. **P1** — multi-tenant is single-tenant in practice. **P1** — all three growth pillars (telemedicine, mobile app, insurance adjudication) are unbuilt; the MOU pilot needs a partner portal + telemed + mobile app, none started → re-baseline the roadmap to solo-dev velocity.
- **P2** — observability inactive + aging beta auth; strategy docs drift toward aspirational-as-status. *(Dismissed: "application backbone unprovisioned is a P1 defect" — refuted; it's honestly-labelled future-phase roadmap work.)*

### Testing & Quality Gates — 38

- **P1** — Kashier webhook untested. **P1** — all EHR server actions (the authz/IDOR surface) untested. **P2** — visit state machine untested; auth flows untested; no npm-audit CI gate; no coverage threshold / no E2E / audit-chain verifier not in CI; tenant guard is a heuristic ratchet, not a cross-tenant proof.

### Observability, Ops & DR — 31 (weakest)

- **P1** — observability inactive in prod. **P1** — single-VPS SPOF + unimplemented/undrilled backups/DR. **P1** — logs console-only, never aggregated. **P2** — no external health endpoint; no durable job queue; document memory-buffering availability risk; **local dev points `DATABASE_URL` at the shared PROD Postgres** (no environment isolation — a stray migration/seed can corrupt live patient data).

## 6. What's genuinely excellent (credit where due)

For a solo founder, several things here are *better* than most funded teams ship:
- A real, enforced **layered authorization engine** — not role-string checks scattered around.
- **IDOR systematically prevented** — IDs re-resolved server-side, case-scope from live CareTeam.
- **Correct FHIR clinical modeling** — proper LOINC/UCUM coding, note-signing with Provenance, `entered-in-error` soft-delete discipline.
- **Perfect EN/AR i18n parity** and a fully-swept RTL/logical-CSS system with CI ratchets.
- **Correct payment crypto** — constant-time HMAC, fail-closed, idempotent transactional ledger.
- **Honest engineering culture** — the strategy docs include a self-correcting "audit revision" admitting prior overstatement. That instinct is worth more than any single feature.

## 7. The action plan (next two quarters)

### Gate 0 — Before the hospital partner goes live (do not launch without these)
1. **Fix bus factor:** hire engineer #2, turn on PR review, fix git identity, commit IaC (compose + Caddyfile + env templates), run one restore drill and record RTO/RPO.
2. **Turn on observability:** set `SENTRY_DSN` (staging + prod), add `withSentryConfig`, stand up an external uptime monitor + an unauthenticated `/api/health`, alert on the audit-failure log markers. *(Small — the seam already exists.)*
3. **Close the audit-integrity gap:** route PHI-download, login/logout, restricted-read, and **all financial** audit rows through `recordAudit` (hash-chained + FHIR-mirrored); wire `verify-audit-chain` into CI.
4. **Authenticate the payment routes** (`initiate`, `instapay`) and make InstaPay metadata write-once.
5. **Fix money integrity:** persist the actual gateway amount; tighten reconciliation to `[1]`; add webhook replay/nonce defense + refund status guard.
6. **Lock the visit state machine:** transition-legality map + single transaction + conditional update; gate `force=on` behind role/break-glass (especially check-out).
7. **Rate-limit login** + enforce suspension/role-downgrade mid-session.
8. **Stop dev pointing at prod DB**; move backups to off-host with alerting.

### Quarter 2 — Harden for scale
- Adopt (or delete) the tenant-scoping client; drive the tenant baseline to zero; add Postgres RLS.
- Change regulatory-table FKs to `Restrict`; add `Patient.nationalId/phone` uniqueness.
- Fix the patient-directory pagination/case-scope bug and the patient-detail N+1; add app-layer caching + a Prisma pool limit.
- Replace the default PWA cache list to exclude PHI; fix `<html>` lang/dir + skip-links.
- Extract the shared clinician write-wrapper (closes the `encounterId` divergence).
- Write tests for the webhook, EHR server-action authz, and the state machine; add the npm-audit CI gate + Renovate.
- Implement DPL consent capture + a data-subject export path; correct the compliance docs to a "claim → evidence" standard.

### Later — Strategy & growth
- Re-baseline the roadmap to real velocity; serve the MOU with a **web-only** partner portal + an embedded Daily.co iframe first (defer the Expo app).
- Plan the Prisma 5→6→7 and next-auth GA upgrades; drop jQuery/react-slick; move `ts-node` to devDeps.
- Implement or remove `ProviderPayout`; make currency a first-class column ahead of MENA.
- Separate Medplum + Postgres from the app host during the OVH migration.
